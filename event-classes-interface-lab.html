You will learn how to design proper classes with properties, how events let parts of your program react to things that happen, and how interfaces help you write flexible, testable code.

📦 Modules covered: 8, 9 ⏱ Duration: ~90 minutes 🛠 Tool: Visual Studio 2022 🎯 Builds on: Labs 1–3
MODULE 8

Events & Delegates

1 What is a delegate?

A delegate is a variable that holds a reference to a method — just like a variable can hold an int or a string, a delegate holds a method. This lets you pass methods around as arguments or store them for later.

Think of it like a contact saved in your phone. The contact (delegate) stores who to call, and when you press "call" (invoke) it runs that method.

Defining and using a delegate 
// 1. Define a delegate type — what signature of method can it hold?
//    This one holds any method that takes a string and returns void
delegate void NotifyHandler(string message);

// 2. Create methods that match the signature
static void SendEmail(string message)
{
    Console.WriteLine("Email sent: " + message);
}

static void LogToConsole(string message)
{
    Console.WriteLine("Log: " + message);
}

// 3. Use the delegate — inside Run()
NotifyHandler notify = SendEmail;   // point to SendEmail
notify("Student S001 enrolled");     // calls SendEmail

notify = LogToConsole;              // now point to LogToConsole
notify("Student S001 enrolled");     // calls LogToConsole

// A delegate can hold MULTIPLE methods at once (multicast)
notify = SendEmail;
notify += LogToConsole;            // add a second method
notify("Student S002 enrolled");   // calls BOTH methods
Output
Email sent: Student S001 enrolled
Log: Student S001 enrolled
Email sent: Student S002 enrolled
Log: Student S002 enrolled
ℹ

In modern C# you rarely define your own delegate type. Instead you use the built-in Action (void return) and Func<T> (returns a value). But understanding delegates first makes events much easier to understand.

2 Events — the publisher/subscriber pattern

An event is a delegate with a restriction: only the class that owns the event can fire it. Other classes can only subscribe to it (add their method) or unsubscribe. This is called the publisher/subscriber pattern.

Real-world analogy: a YouTube channel (publisher) posts a video. Subscribers get notified automatically. The channel does not need to know who subscribed — it just fires the event.

A class with an event — write outside Module8 class 
// The publisher — fires events when something happens
class EnrollmentService
{
    // Declare the event using a delegate type
    public event Action<string> OnStudentEnrolled;

    public void Enroll(string studentName)
    {
        Console.WriteLine("Enrolling: " + studentName);

        // Fire the event — notify all subscribers
        // The ?. means: only fire if someone is subscribed
        OnStudentEnrolled?.Invoke(studentName);
    }
}
3 Subscribing to an event

Any other code can subscribe to the event using +=. When the event fires, every subscribed method runs automatically. The publisher does not know or care which methods subscribed — it just fires.

Subscribing and unsubscribing — inside Run()
EnrollmentService service = new EnrollmentService(); // Subscribe — add methods that should run when the event fires service.OnStudentEnrolled += SendWelcomeEmail; service.OnStudentEnrolled += LogEnrollment; // Enroll a student — both methods run automatically service.Enroll("Raushan"); Console.WriteLine(); // Unsubscribe one handler service.OnStudentEnrolled -= SendWelcomeEmail; // Now only LogEnrollment runs service.Enroll("Rahul"); // ────────────────────────────────────── // The handler methods (write inside the class) static void SendWelcomeEmail(string name) { Console.WriteLine("Email sent to: " + name); } static void LogEnrollment(string name) { Console.WriteLine("Logged: " + name + " enrolled"); }
Output
Enrolling: Raushan
Email sent to: Raushan
Logged: Raushan enrolled

Enrolling: Rahul
Logged: Rahul enrolled
💡

Notice that EnrollmentService does not know about email or logging at all. You can add new behaviours (SMS, push notification) just by subscribing more methods — without changing the service class. This is the power of event-driven programming.

MODULE 9

Creating Classes & Interfaces

1 Designing a reusable class

A class is a blueprint for creating objects. It bundles related data (fields) and behaviour (methods) together. When you create an object from a class using new, you get an instance — your own copy of that blueprint with its own data.

A Student class — write outside Module9 class
class Student { // Fields — the data this class holds public string Id; public string Name; public int Age; public int Marks; // Constructor — runs when you create a new Student public Student(string id, string name, int age, int marks) { Id = id; Name = name; Age = age; Marks = marks; } // Method — behaviour that belongs to a student public string GetGrade() { if (Marks >= 90) return "A"; if (Marks >= 75) return "B"; if (Marks >= 50) return "C"; return "F"; } public void Print() { Console.WriteLine(Id + " | " + Name + " | Grade: " + GetGrade()); } } // Using it inside Run() Student s1 = new Student("S001", "Raushan", 28, 88); Student s2 = new Student("S002", "Rahul", 24, 65); s1.Print(); // S001 | Raushan | Grade: B s2.Print(); // S002 | Rahul | Grade: C // Objects are independent — changing s1 does not affect s2 s1.Marks = 95; Console.WriteLine(s1.GetGrade()); // A Console.WriteLine(s2.GetGrade()); // C (unchanged)
2 Properties — controlled access to data

Public fields let anyone change data freely — even to invalid values. Properties let you add logic when data is read or written. This protects your object's data.

Using properties instead of public fields
class Course { // Private backing field — only this class can touch it directly private int _durationDays; // Public property — controlled access public int DurationDays { get { return _durationDays; } set { if (value < 1) throw new Exception("Duration must be at least 1 day"); _durationDays = value; } } // Auto-property — shorthand when no validation is needed public string Name { get; set; } // Read-only auto-property public string Code { get; } public Course(string code, string name, int days) { Code = code; Name = name; DurationDays = days; // goes through the property setter } } // Using it Course c = new Course("AZ-204", "Azure Developer", 5); Console.WriteLine(c.Name + " - " + c.DurationDays + " days"); // This would throw an exception — validation in the setter // c.DurationDays = 0;
💡

In most classes you will use auto-properties (public string Name { get; set; }) for simplicity. Add the full get/set with a backing field only when you need validation or computed values.

3 Interfaces — defining a contract

An interface defines a contract — a list of methods and properties that a class promises to provide. It has no code, only signatures. Any class that implements the interface must provide the actual code for all those members.

This is powerful because your code can work with the interface type instead of a specific class — making it easy to swap implementations later.

Defining an interface and two implementations
// Interface — what any notifier MUST be able to do interface INotifier { void Send(string recipient, string message); } // Implementation 1 — email class EmailNotifier : INotifier { public void Send(string recipient, string message) { Console.WriteLine("Email to " + recipient + ": " + message); } } // Implementation 2 — SMS class SmsNotifier : INotifier { public void Send(string recipient, string message) { Console.WriteLine("SMS to " + recipient + ": " + message); } } // Using the interface — inside Run() // The variable type is INotifier — works with any implementation INotifier notifier = new EmailNotifier(); notifier.Send("Raushan", "Your enrollment is confirmed."); // Swap to SMS — same code, different behaviour notifier = new SmsNotifier(); notifier.Send("Raushan", "Your enrollment is confirmed.");
Output
Email to Raushan: Your enrollment is confirmed.
SMS to Raushan: Your enrollment is confirmed.
A method that works with any INotifier
// This method does not care whether it's email, SMS, or anything else static void NotifyStudent(INotifier notifier, string name, string msg) { notifier.Send(name, msg); } // Works with either implementation NotifyStudent(new EmailNotifier(), "Raushan", "Welcome!"); NotifyStudent(new SmsNotifier(), "Rahul", "Welcome!");
ℹ

By convention, interface names in C# start with a capital I — INotifier, IRepository, ILogger. This makes them instantly recognisable when reading code.

4 SOLID — introduction

SOLID is a set of 5 principles that help you write code that is easier to maintain and extend. You do not need to know all five deeply yet — but you should recognise the two you have already been practising.

The 5 SOLID principles — overview
S — Single Responsibility Principle A class should do ONE thing. Bad: class Student handles data + validation + email + database Good: Student holds data, Validator validates, EmailService sends email O — Open/Closed Principle Open for extension, closed for modification. Add new behaviour by adding new classes (like new INotifier implementations) not by editing existing working code. L — Liskov Substitution Principle If a method accepts INotifier, any implementation should work correctly. You already did this — EmailNotifier and SmsNotifier are interchangeable. I — Interface Segregation Principle Don't put too many methods in one interface. Many small focused interfaces are better than one large one. D — Dependency Inversion Principle Depend on interfaces, not on concrete classes. NotifyStudent(INotifier notifier) is better than NotifyStudent(EmailNotifier e) because it works with any notifier — not just email.
💡

You have already applied S (each class has one job), L (INotifier works with any implementation), and D (NotifyStudent accepts INotifier, not EmailNotifier). SOLID is not theory — it shows up naturally when you write clean code.
