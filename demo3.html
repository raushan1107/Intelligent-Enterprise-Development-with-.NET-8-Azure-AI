01 · Concepts
Control Flow — What You Must Know
🔀
Switch Expression (C# 8+)
Returns a value — it's an expression, not a statement. Each arm is pattern => result. The _ discard is the mandatory catch-all. Compiler checks exhaustiveness — miss a case and you get a warning at build time.
string grade = score switch { >= 90 => "A+", >= 80 and < 90 => "A", >= 60 => "B/C", _ => "F" };
🔁
foreach — Most Common Loop
Use foreach for 90% of iterations — collections, arrays, LINQ results. Use for only when you need the index for something beyond element access. Prefer continue to skip over bad data instead of wrapping in an if.
foreach (var s in students) { if (string.IsNullOrEmpty(s.Name)) continue; // skip, next iteration Process(s); }
🔄
while & do-while
while — check condition first, may never run. do-while — always runs once, check after. Use do-while for input loops where you must read before checking. Use while for retry loops with an attempt counter.
// do-while: perfect for input menus do { Console.Write("Enter ID: "); input = Console.ReadLine(); Process(input); } while (input != "quit");
💥
try / catch / finally
Surround risky code in try. Catch blocks are checked top-down — most specific first. finally always runs — perfect for cleanup. Never catch Exception and swallow it silently. Always log or rethrow.
try { var n = int.Parse(input); } catch (FormatException fe) { // specific Log(fe.Message); } catch (Exception ex) { // catch-all last Log(ex); throw; // rethrow! } finally { conn.Close(); // always runs }
🎯
when — Exception Filters
The when clause runs BEFORE the catch block. If it returns false, the catch is skipped entirely — and crucially, the stack is NOT unwound yet, so the debugger sees the original call site. Ideal for SQL error codes, HTTP status codes.
catch (SqlException ex) when (ex.Number == 1205) { // Only catches deadlock (1205) // Stack NOT unwound if filter = false RetryOperation(); }
🔄
throw vs throw ex
Always use bare throw to re-throw — it preserves the full original stack trace. throw ex resets the stack trace to the catch block, hiding where the exception actually originated. This makes production debugging much harder.
catch (Exception ex) { Log(ex); throw; // ✓ preserves stack trace // throw ex; ❌ resets to HERE — hides origin }

// ── BASIC SWITCH EXPRESSION ── string GetGrade(int score) => score switch { >= 90 => "A+", >= 80 and < 90 => "A", >= 70 => "B", >= 60 => "C", >= 40 => "D", _ => "F" // _ = discard (default) }; // ── TYPE PATTERN ── string Describe(object obj) => obj switch { Student s => $"Student: {s.Name}", Course c => $"Course: {c.Code}", null => "(null)", _ => $"Other: {obj.GetType().Name}" }; // ── PROPERTY PATTERN ── decimal GetFeeRate(Student s) => s switch { { GPA: > 3.8m, HasScholarship: true } => 0.50m, { GPA: > 3.0m } => 0.80m, { IsActive: false } => 0.00m, _ => 1.00m }; // ── TUPLE PATTERN ── string GetAction(EligStatus s, bool paid) => (s, paid) switch { (EligStatus.Approved, true) => "Enroll", (EligStatus.Approved, false) => "Collect fee", (EligStatus.Blocked, _) => "Contact admin", _ => "Review" };
Switch Patterns
1
Expression, not statement
Switch expression returns a value — assign it to a variable, pass it as an argument, or return it directly. Every arm must return the same type. Compiler checks all cases are covered.
var g = score switch { ... };
2
Top-down, first match wins
Arms are evaluated top-to-bottom. Put most-specific patterns first. >= 90 must come before >= 80. The _ discard at the bottom catches everything that didn't match.
Most specific → least specific → _
3
Property pattern
Test object properties inline. Combine with and: { GPA: > 3.8m, HasScholarship: true }. Both conditions must be true. No temporary variables needed.
{ GPA: > 3.8m, Active: true }
4
Tuple pattern
Match multiple values at once with a tuple. Each arm tests a combination. Perfect for state machines: (currentState, event) → nextState.
(status, paid) switch { ... }
Loops.cs
// ── foreach — most idiomatic ── var roster = new[] { "Alice", "", "Bob", null, "Riya" }; foreach (var name in roster) { if (string.IsNullOrWhiteSpace(name)) continue; Console.WriteLine($"Enrolling: {name}"); } // Output: Alice, Bob, Riya (skipped "", null) // ── for — when you need the index ── int[] scores = { 85, 72, 91, 63 }; for (int i = 0; i < scores.Length; i++) Console.WriteLine($"Student {i+1}: {scores[i]}"); // ── while — retry loop ── int attempts = 0; bool ok = false; while (attempts < 3 && !ok) { ok = TryRegister(studentId); attempts++; } // ── do-while — input validation ── string? input; do { Console.Write("Enter ID (or 'quit'): "); input = Console.ReadLine(); if (input != "quit") ProcessStudent(input!); } while (input != "quit"); // ── break exits the whole loop ── foreach (var s in students) { if (enrollCount >= MAX) break; // done if (!s.IsEligible) continue; // skip this one Enroll(s); enrollCount++; }
Loop Guide
1
continue skips, break stops
continue — skip the rest of THIS iteration, jump to the next one. break — exit the entire loop immediately. return — exit the whole method (strongest). Use the weakest one that solves the problem.
continue = skip one break = exit loop return = exit method
2
do-while runs at least once
The body ALWAYS executes at least once — the condition is checked AFTER. Perfect for menus and input loops where you must read before you can validate. Cleaner than while(true) + break.
do { read; } while (!valid);
3
for vs foreach
Use for only when you need the numeric index for something beyond just accessing the element. Otherwise foreach is cleaner and avoids off-by-one errors (starting at 0, stopping at Length-1).
for when index matters foreach otherwise
ExceptionHandling.cs
// ── try / catch / finally ── try { var batchSize = int.Parse(Console.ReadLine()!); ProcessBatch(batchSize); } catch (FormatException fe) // most specific first { Console.WriteLine($"Not a number: {fe.Message}"); } catch (ArgumentOutOfRangeException aoore) when (aoore.ParamName == "batchSize") // filter { Console.WriteLine("Batch must be 1–30"); } catch (Exception ex) // catch-all LAST { LogError("FATAL", ex); throw; // preserve stack ✓ } finally { CloseResources(); // ALWAYS runs } // ── using = auto-dispose (like finally) ── await using var conn = new SqlConnection(cs); await conn.OpenAsync(); // DisposeAsync() called automatically — even on exception // ── Guard clauses — throw early ── public void Enroll(Student? s, Course? c) { ArgumentNullException.ThrowIfNull(s); // .NET 7+ ArgumentNullException.ThrowIfNull(c); if (s.Age < 15) throw new ArgumentException( "Min age 15", nameof(s.Age)); // ... happy path below }
Exception Rules
1
Most-specific catch first
Catch blocks are checked top-to-bottom. FormatException must come before Exception. If Exception were first, it would catch EVERYTHING — specific blocks would never run. Compiler won't warn you; it's a logic bug.
FormatException first Exception last
2
when filter — no stack unwind
The when clause runs before the catch block starts. If it's false, the exception continues searching for another handler. The stack is still intact — the debugger can still see the original throw location.
catch (Ex) when (condition)
3
finally always runs
Even if no catch matches, even if there's a return inside the try, finally still runs. The CLR guarantees this. Use for closing connections, releasing locks, logging completion. using is syntactic sugar over try/finally.
finally = guaranteed cleanup
4
Guard clauses — fail fast
Validate inputs at the top of the method and throw immediately if invalid. This is the "fail fast" pattern — better to crash clearly at the entry point than silently produce wrong results 10 lines later.
ArgumentNullException.ThrowIfNull(s)
EnrollmentSession.cs — ERP Context
// Full enrollment session using all M03 concepts public static void RunSession() { Console.WriteLine("=== ERP Enrollment Session ==="); string? input; do // do-while: run at least once { Console.Write("\nStudent ID (or 'done'): "); input = Console.ReadLine()?.Trim(); if (input == "done") break; if (string.IsNullOrEmpty(input)) continue; try { var student = FindStudent(input); var action = CheckEligibility(student) switch { EligStatus.Approved => "✓ Enrolling", EligStatus.FeePending => "⚠ Fee pending", EligStatus.Blocked => "✕ Blocked — see admin", _ => throw new InvalidOperationException( "Unknown status") }; Console.WriteLine($" {action}: {student.Name}"); } catch (StudentNotFoundException ex) { Console.WriteLine($" ⚠ {ex.Message}"); // Recoverable — loop continues } catch (Exception ex) { Console.WriteLine(" ✕ Unexpected error"); LogFatal(ex); throw; // fatal — rethrow, preserve trace } } while (true); Console.WriteLine("\nSession complete."); }
ERP Patterns
1
do-while(true) + break
The REPL (Read-Eval-Print Loop) pattern. Runs forever until the user types "done". Using do-while(true) is clearer than while(true) because it shows intent: this always executes once before any exit check.
do { } while (true) + break on "done"
2
continue skips empty input
User pressed Enter without typing. continue skips the rest of this iteration and goes back to the top of the loop — prompting again. Much cleaner than wrapping the entire loop body in if (!empty).
if (empty) continue; // re-prompt
3
Business vs fatal exceptions
StudentNotFoundException is expected — show message and continue. Unknown Exception is fatal — log and rethrow. Two-tier error handling: recover from domain errors, crash-fast on unexpected ones.
domain error → recover unexpected → rethrow
4
throw in switch default arm
The _ => throw new InvalidOperationException() ensures that if an unknown EligStatus value appears, it fails immediately with a clear message — rather than silently doing nothing. Fail loudly on programmer errors.
_ => throw new InvalidOp(...)

07 · Watch Out
Common Mistakes in M03
⚠️
Control Flow Mistakes That Cause Real Bugs
❌
Catch blocks in wrong order: Putting catch (Exception ex) before catch (FormatException fe). The general Exception swallows everything — FormatException never runs. Fix: Always put most-specific exceptions FIRST. Exception should always be last.
❌
Using throw ex instead of throw: catch (Exception ex) { throw ex; } — resets the stack trace to the catch block. The original call site is hidden, making production debugging very hard. Fix: catch (Exception ex) { Log(ex); throw; } // bare throw preserves full trace
❌
Swallowing exceptions silently: catch (Exception) { } — empty catch block. The error disappears. Your app continues with corrupted state. This is one of the worst patterns in professional code. Fix: Always at minimum: catch (Exception ex) { Log(ex); throw; }
❌
Missing the _ discard in switch: If you add a new enum value later and forget to add it to the switch, the switch silently falls through with no warning. Fix: Always end with: _ => throw new InvalidOperationException($"Unhandled: {value}")
❌
Using int.Parse() on user input: If the user types letters, int.Parse(Console.ReadLine()!) throws an unhandled FormatException and crashes. Fix: if (int.TryParse(Console.ReadLine(), out int n)) { ... } else { show error }
