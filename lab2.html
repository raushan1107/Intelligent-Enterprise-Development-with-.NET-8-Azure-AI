Methods, Exceptions & Logging

You will learn how to break your code into reusable methods, handle errors properly, and understand how real applications record what they are doing.

📦 Modules covered: 4, 5 ⏱ Duration: ~90 minutes 🛠 Tool: Visual Studio 2022 🎯 Builds on: Lab 1
MODULE 4

Methods & Overloading

1 What is a method?

A method is a named block of code that does one job. You define it once and call it as many times as you need. This avoids copying and pasting the same code in multiple places.

Think of it like a machine: you give it some input, it does something, and it may give you back a result.

Defining and calling a simple method 
// DEFINING a method — write this outside of Run(), at the bottom of the class
static void PrintWelcome()
{
    Console.WriteLine("Welcome to the .NET 8 course!");
}

// CALLING the method — write this inside Run()
PrintWelcome();   // call it once
PrintWelcome();   // call it again — same code, no repetition
Output
Welcome to the .NET 8 course!
Welcome to the .NET 8 course!
ℹ

void means the method does not return a value — it just does something. static means you can call it without creating an object first. You'll learn about objects in a later module.

2 Return types

When a method gives you back a result, you specify the type of that result instead of void. The method must use the return keyword to send the value back.

Methods with return values 
// Returns an int
static int AddNumbers(int a, int b)
{
    return a + b;
}

// Returns a string
static string GetFullName(string first, string last)
{
    return first + " " + last;
}

// Returns a bool
static bool IsAdult(int age)
{
    return age >= 18;
}

// Calling them and storing the result in a variable
int    total    = AddNumbers(10, 20);
string fullName = GetFullName("Raushan", "Ranjan");
bool   canVote  = IsAdult(20);

Console.WriteLine(total);     // 30
Console.WriteLine(fullName);  // Raushan Ranjan
Console.WriteLine(canVote);   // True
✦ Try It
  • Write a method called Multiply that takes two ints and returns their product.
  • Write a method called IsEven that takes an int and returns true if it is even.
  • Call both methods and print the results.
3 Parameters — passing data into a method

Parameters are the inputs your method needs to do its job. You list them inside the parentheses when defining the method.

Method with parameters 
static void PrintCourse(string name, int days)
{
    Console.WriteLine("Course: " + name + " | Days: " + days);
}

// Each call passes different values
PrintCourse("C# Basics", 3);
PrintCourse("Azure AI", 5);
PrintCourse("Blazor", 2);
Output
Course: C# Basics | Days: 3
Course: Azure AI | Days: 5
Course: Blazor | Days: 2
💡

Parameters are local to the method. Changing a parameter value inside a method does not affect the variable you passed in from outside. This is called pass by value.

4 Default parameter values

You can give a parameter a default value. If the caller does not pass that argument, the default is used automatically. This makes parameters optional.

Optional parameters with defaults 
static void PrintInvoice(string invoiceId,
                           string studentName = "Guest",
                           int    amount     = 0)
{
    Console.WriteLine(invoiceId + " | " + studentName + " | " + amount);
}

PrintInvoice("INV-001");                      // uses both defaults
PrintInvoice("INV-002", "Raushan");           // uses default amount
PrintInvoice("INV-003", "Rahul", 2999);      // all values provided
Output
INV-001 | Guest | 0
INV-002 | Raushan | 0
INV-003 | Rahul | 2999
✦ Try It
  • Write a method Greet(string name, string greeting = "Hello") and call it both with and without the greeting argument.
  • What happens if you try to make the first parameter optional but not the second? Try it and read the compiler error.
5 Method overloading

Overloading means having multiple methods with the same name but different parameters. The compiler decides which one to call based on what you pass.

This is useful when the same action makes sense for different types of input — you don't need to invent different names like ShowInfoByName, ShowInfoByNameAndAge, etc.

Three overloads of ShowInfo 
// Overload 1 — just a name
static void ShowInfo(string name)
{
    Console.WriteLine("Name: " + name);
}

// Overload 2 — name and age
static void ShowInfo(string name, int age)
{
    Console.WriteLine("Name: " + name + " | Age: " + age);
}

// Overload 3 — name, age, and city
static void ShowInfo(string name, int age, string city)
{
    Console.WriteLine("Name: " + name + " | Age: " + age + " | City: " + city);
}

// Calling them — compiler picks the right version automatically
ShowInfo("Raushan");
ShowInfo("Raushan", 28);
ShowInfo("Raushan", 28, "Noida");
Output
Name: Raushan
Name: Raushan | Age: 28
Name: Raushan | Age: 28 | City: Noida
ℹ

Overloads must differ in the number or types of parameters. Changing only the return type is not enough — the compiler cannot tell them apart that way.

✦ Try It
  • Create two overloads of a method called Describe — one that takes a string, one that takes an int. Call both.
  • Try creating two methods with the same name and same parameters but different return types — read the error the compiler gives you.
6 Refactoring for reusability

Refactoring means moving repeated code into a method. The program does exactly the same thing — but the code is cleaner and easier to maintain. If the calculation ever changes, you fix it in one place.

Before — repeated logic
double fee1 = 2000 + (2000 * 0.18);
double fee2 = 3000 + (3000 * 0.18);
double fee3 = 1500 + (1500 * 0.18);
// Same formula 3 times.
// If GST changes to 20%,
// you must fix 3 lines.
After — extracted method
double fee1 = AddGST(2000);
double fee2 = AddGST(3000);
double fee3 = AddGST(1500);
// Fix GST rate in one
// method — all calls
// update automatically.
The extracted method 
static double AddGST(double baseAmount)
{
    return baseAmount + (baseAmount * 0.18);
}

Console.WriteLine(AddGST(2000));   // 2360
Console.WriteLine(AddGST(3000));   // 3540
Console.WriteLine(AddGST(1500));   // 1770
✦ Try It
  • Look at your Lab 1 exercise code. Is there any repeated code you can move into a method? Try refactoring it.
  • Write a method PrintSeparator() that prints a line of dashes. Call it between sections of your output to make it readable.
MODULE 5

Exception Handling & Logging

1 Multiple catch blocks

You saw basic try/catch in Lab 1. In real programs, different things can go wrong and each type of error needs a different response. You can stack multiple catch blocks — C# tries them from top to bottom and runs the first one that matches.

Rule: always put the most specific exception types first, and put the general Exception catch last as a safety net.

Multiple catch blocks 
string input = "abc";

try
{
    int number = int.Parse(input);
    Console.WriteLine("Parsed: " + number);
}
catch (FormatException ex)
{
    // runs when the string is not a valid number
    Console.WriteLine("Not a valid number: " + ex.Message);
}
catch (OverflowException ex)
{
    // runs when the number is too large for an int
    Console.WriteLine("Number too large: " + ex.Message);
}
catch (Exception ex)
{
    // catches anything else — always last
    Console.WriteLine("Unexpected error: " + ex.Message);
}
Output
Not a valid number: Input string was not in a correct format.
✦ Try It
  • Change input to "999999999999" — which catch block runs now?
  • Swap the order of the catch blocks so Exception comes first. Run it — does Visual Studio warn you? What happens?
2 throw — raising an exception yourself

You don't have to wait for C# to throw an exception. You can throw one yourself when your code detects an invalid situation — for example, when a user provides bad data.

Using throw to validate input 
static void SetAge(int age)
{
    if (age < 0 || age > 120)
    {
        // throw stops the method and sends the error to the caller
        throw new Exception("Age " + age + " is not valid.");
    }
    Console.WriteLine("Age set to: " + age);
}

// Calling it
try
{
    SetAge(-5);    // this will throw
}
catch (Exception ex)
{
    Console.WriteLine("Caught: " + ex.Message);
}
Output
Caught: Age -5 is not valid.
💡

When you throw, execution of the current method stops immediately. C# looks for the nearest catch block that matches, working its way up through all the methods that were called to get here.

✦ Try It
  • Add validation to a method that accepts a course fee — throw if the fee is negative.
  • Call the method with a valid value and then with an invalid value. Confirm both cases work correctly.
3 Custom exceptions

Instead of throwing a generic Exception, you can create your own exception class. This gives the error a meaningful name that makes sense in your application — the caller can then catch your specific type and handle it differently from other errors.

A custom exception is just a class that inherits from Exception.

Step 1 — Define your custom exception class 
// Write this outside of your Module5 class, at the bottom of the file
class EnrollmentException : Exception
{
    public EnrollmentException(string message) : base(message)
    {
        // base(message) passes your message to the Exception class
        // so that ex.Message will return it
    }
}
Step 2 — Throw and catch your custom exception 
static void EnrollStudent(string studentId, bool isActive)
{
    if (!isActive)
    {
        throw new EnrollmentException(
            "Student " + studentId + " is not active.");
    }
    Console.WriteLine("Enrolled: " + studentId);
}

// Calling it
try
{
    EnrollStudent("S001", isActive: false);
}
catch (EnrollmentException ex)
{
    // catches only EnrollmentException, not other errors
    Console.WriteLine("Enrollment failed: " + ex.Message);
}
Output
Enrollment failed: Student S001 is not active.
✦ Try It
  • Create a second custom exception called PaymentException.
  • Write a method ProcessPayment(decimal amount) that throws PaymentException if amount is zero or negative.
  • Call it and catch the specific exception.
4 Logging fundamentals

Logging means recording what your application is doing as it runs. When something goes wrong in production, logs are often the only way to find out what happened.

Every log entry has a level that tells you how serious the message is:

Log levels — from least to most serious
DEBUG    → detailed info for developers, turned off in production
INFO     → normal events (user logged in, payment processed)
WARNING  → something unusual happened but the app still works
ERROR    → something failed and needs attention
CRITICAL → the system itself is affected (DB down, disk full)
Simulating a logger with a simple method 
// A simple helper to print log messages consistently
static void Log(string level, string message)
{
    string time = DateTime.Now.ToString("HH:mm:ss");
    Console.WriteLine("[" + level + "] " + time + " - " + message);
}

// Using the logger
Log("DEBUG",    "Checking student record for S001");
Log("INFO",     "Student S001 enrolled in Azure AI");
Log("WARNING",  "Course is almost full (28 of 30 seats)");
Log("ERROR",    "Payment failed for invoice INV-101");
Log("CRITICAL", "Database connection lost");
Output
[DEBUG]    14:32:01 - Checking student record for S001
[INFO]     14:32:01 - Student S001 enrolled in Azure AI
[WARNING]  14:32:01 - Course is almost full (28 of 30 seats)
[ERROR]    14:32:01 - Payment failed for invoice INV-101
[CRITICAL] 14:32:01 - Database connection lost
ℹ

In real .NET projects you use Microsoft.Extensions.Logging or Serilog — they work exactly like this Log() method but write to files, databases, or cloud services like Azure Application Insights. The concept is the same.

Logging inside try/catch — the real-world pattern 
try
{
    Log("INFO", "Attempting to enroll student S002");
    EnrollStudent("S002", isActive: false);
}
catch (EnrollmentException ex)
{
    Log("ERROR", "Enrollment failed: " + ex.Message);
}
finally
{
    Log("DEBUG", "Enrollment attempt finished");
}
✦ Try It
  • Add the Log() method to your project and call it from inside a try/catch block.
  • Try logging before the try, inside catch, and in finally — observe the order of messages in the output.
5 Introduction to structured logging

Traditional logging builds a single string by joining text together. Structured logging keeps each piece of data separate so you can search and filter logs later.

Traditional — flat string
"Student S001 enrolled in AZ-204"

// Just text — cannot filter
// by StudentId or CourseCode
// in a log viewer tool
Structured — template + values
"Student {StudentId} enrolled in {CourseCode}"
// Values stored separately:
// StudentId = "S001"
// CourseCode = "AZ-204"
// → queryable in Serilog/Seq
Why structured logging matters 
string studentId = "S001";
string course    = "AZ-204";
int    seat      = 14;

// Traditional — one flat string, hard to search
Console.WriteLine("Student " + studentId + " enrolled in " + course);

// Structured template — each value is a named property
// In Serilog you write this as:
// _logger.LogInformation("Student {StudentId} enrolled in {Course} seat {Seat}",
//                         studentId, course, seat);
//
// The log tool stores: StudentId="S001", Course="AZ-204", Seat=14
// You can then query: show all logs WHERE StudentId = "S001"

// For now, simulate it:
Console.WriteLine("Template : Student {StudentId} enrolled in {Course} seat {Seat}");
Console.WriteLine("Values   : StudentId=" + studentId +
                 " Course=" + course + " Seat=" + seat);
💡

In Day 5 when you add Serilog to the project, you will write real structured logging with almost no extra effort. The concept you learned here is exactly what the library implements under the hood.

✦ Try It
  • Write a structured log message for a payment event with fields: StudentId, Amount, InvoiceId.
  • Print both the template string and the individual values — as if a log viewer would store them separately.
