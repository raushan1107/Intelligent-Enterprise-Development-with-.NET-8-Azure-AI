01 · Concepts
What You Need to Know
🌐
.NET 8 — Runtime + SDK
The SDK is your development kit — compiler, tools, project templates. The Runtime is what end users install to run your app. LTS (Long-Term Support) = security patches until November 2026. Always target net8.0 for new projects.
dotnet --version // 8.x.x dotnet --list-runtimes // installed runtimes dotnet --list-sdks // installed SDKs
🔬
Roslyn — The C# Compiler
Roslyn reads your .cs files, builds an AST (Abstract Syntax Tree), runs type-checking, and emits IL (Intermediate Language) into a .dll. Errors like "CS0029: Cannot convert string to int" come from Roslyn — before your code ever runs.
// Roslyn catches this at BUILD time: int x = "hello"; // CS0029 ← compile error // You never run bad-typed code
⚡
JIT — Just-In-Time Compiler
When your app runs, the CLR's JIT compiles each method from IL to native CPU instructions the first time it's called. Tiered compilation: Tier 0 = fast (quick startup), Tier 1 = optimised (hot path). Subsequent calls use the cached native code.
// .NET 8 also supports Native AOT: // dotnet publish --aot // → no JIT, instant startup, smaller size // Used for serverless / containers
🗑️
Garbage Collector (GC)
The GC manages heap memory — you never call free(). It uses generations: Gen0 (new, collected often), Gen1, Gen2 (long-lived). Most objects die in Gen0 — collections are fast. The GC only frees memory; IDisposable frees other resources (files, connections).
// GC frees MEMORY automatically. // IDisposable frees RESOURCES manually. await using var conn = new SqlConnection(cs); // conn auto-disposed at end of scope ✓
📁
SDK-Style .csproj
The project file is minimal XML — no file lists, no GUIDs. The SDK auto-includes all .cs files. Key properties: TargetFramework, Nullable, ImplicitUsings. Solution files (.sln) group multiple projects together.
<Project Sdk="Microsoft.NET.Sdk"> <PropertyGroup> <TargetFramework>net8.0</TargetFramework> <Nullable>enable</Nullable> <ImplicitUsings>enable</ImplicitUsings> </PropertyGroup> </Project>
⌨️
Top-Level Statements
.NET 6+ allows Program.cs without a class or Main method. The compiler generates the entry point. Only ONE file per project can use top-level statements. ImplicitUsings auto-adds System, System.Linq, etc.
// Old way (still valid): class Program { static void Main(string[] args) { ... } } // New way (.NET 6+): Console.WriteLine("Hello!"); // just this

// .NET 8 top-level statements // No class Program, no static void Main // The compiler generates the entry point for you. // ImplicitUsings=enable means these are auto-added: // using System; // using System.Collections.Generic; // using System.Linq; ... and more Console.WriteLine("🏫 ERP Enrollment System v1.0"); // var = local type inference (still strongly typed) var appName = "ERP Core"; // inferred: string const int MAX = 30; // compile-time constant var version = 1.0; // inferred: double Console.WriteLine($"\n{appName} — Max Batch: {MAX}"); // List<string> — generic collection var roster = new List<string> { "Alice Sharma", "Bob Kapoor", "Riya Patel" }; Console.WriteLine($"Enrolled: {roster.Count} / {MAX}"); foreach (var name in roster) Console.WriteLine($" ↳ {name}"); Console.WriteLine("\nSession complete. Press any key..."); Console.ReadKey();
Line-by-Line
1
No class / no Main
Top-level statements (C# 10+). The Roslyn compiler wraps everything in a generated static async Task Main(string[] args). You can still access args as a built-in variable.
// args is available automatically
2
var — type inference
var is NOT dynamic. The type is resolved at compile time by Roslyn. var appName = "ERP" → type is string. You get full IntelliSense and type safety.
var x = "hi"; // string, not dynamic
3
const int MAX = 30
Compile-time constant — the value 30 is baked directly into IL. Zero runtime overhead. Can only be primitives or string. Use for fixed business rules: max batch size, fee rate, etc.
const decimal GST = 0.18m;
4
$"" Interpolated String
Evaluated left-to-right. The compiler converts it to a String.Format call or a DefaultInterpolatedStringHandler (faster, .NET 6+). Format specifiers work inside: {fee:C2} for currency.
$"{fee:C2}" // ₹ 15,000.00
5
Console.ReadKey()
Keeps the console window open by waiting for a keypress. Use Ctrl+F5 in VS to run without debugger — it keeps the window open automatically without needing ReadKey.
Console.ReadKey(); // wait for input
ERP.Enrollment.csproj
<!-- SDK-style project — minimal XML --> <!-- The SDK auto-includes ALL .cs files --> <Project Sdk="Microsoft.NET.Sdk"> <PropertyGroup> <!-- Target framework --> <OutputType>Exe</OutputType> <TargetFramework>net8.0</TargetFramework> <!-- Nullable reference types (recommended) --> <Nullable>enable</Nullable> <!-- Auto-add common using statements --> <ImplicitUsings>enable</ImplicitUsings> <!-- App metadata (optional) --> <AssemblyName>ERP.Enrollment</AssemblyName> <RootNamespace>ERP.Enrollment</RootNamespace> <!-- Treat warnings as errors in Release --> <TreatWarningsAsErrors Condition="'$(Configuration)'=='Release'"> true </TreatWarningsAsErrors> </PropertyGroup> </Project>
Project File Explained
1
OutputType: Exe vs Library
Exe = runnable application (has an entry point). Library = class library with no entry point (used as a dependency). ASP.NET Core apps use Exe even though they're web apps.
Exe = runnable | Library = no main
2
Nullable=enable
Activates nullable reference type analysis. string name cannot be null — the compiler warns if you try. string? name explicitly allows null. Prevents 60–70% of NullReferenceExceptions at compile time.
string name = null; // CS8600 warning string? name = null; // OK ✓
3
TreatWarningsAsErrors
In Release builds, any compiler warning becomes a build error. This forces your CI/CD pipeline to fail if warning-level issues exist. A warning in dev is a bug in production.
Condition="Release" // only in prod builds
Adding NuGet Packages
// METHOD 1: CLI (recommended for precision) dotnet add package Serilog --version 3.1.1 dotnet add package Serilog.Sinks.Console --version 5.0.1 dotnet add package Serilog.Sinks.File --version 5.0.0 // METHOD 2: VS 2022 NuGet Manager // Right-click project → Manage NuGet Packages // Browse tab → search "Serilog" → Install // What gets added to .csproj automatically: <ItemGroup> <PackageReference Include="Serilog" Version="3.1.1" /> <PackageReference Include="Serilog.Sinks.Console" Version="5.0.1" /> </ItemGroup> // RESTORE (happens automatically on build): // dotnet restore // Packages cached at: ~/.nuget/packages/ // Never commit the packages folder to Git! // CHECK OUTDATED: // dotnet list package --outdated
NuGet Essentials
1
NuGet = .NET's npm
NuGet is the package manager for .NET — like npm for Node.js or pip for Python. Over 600,000 packages at nuget.org. Packages are versioned DLLs distributed as .nupkg files.
nuget.org → search → add reference
2
Pin versions exactly
Always pin major versions to avoid unexpected breaking changes. Use --version 3.1.1 not just dotnet add package Serilog (which gets latest — could be a major version bump).
Version="3.1.1" ← exact pin
3
.gitignore packages
Never commit the packages/ or .nuget/ folders to Git. The .csproj references are enough — dotnet restore downloads everything. The default .gitignore from GitHub already handles this.
dotnet restore # re-downloads all packages
GlobalUsings.cs (create this file)
// Create this file: ERP.Enrollment/GlobalUsings.cs // These usings apply to EVERY .cs file in the project. // No need to repeat them in each file. // ImplicitUsings=enable already adds: // using System; // using System.Collections.Generic; // using System.IO; // using System.Linq; // using System.Net.Http; // using System.Threading; // using System.Threading.Tasks; // Add YOUR app-wide usings here: global using Microsoft.Extensions.Logging; global using ERP.Enrollment.Models; global using ERP.Enrollment.Services; global using ERP.Enrollment.Exceptions; // Result: every other .cs file in this project // has access to all these namespaces // WITHOUT any using statement at the top. // This replaces 5-8 using lines per file. // In a 50-file project, that's 250+ lines saved.
Global Usings (C# 10)
1
global using = project-wide
A global using applies the namespace to every file in the project. No performance cost — this is purely a compile-time directive. The IL output is identical to adding the using in every file.
global using System.Text.Json;
2
Convention: one file
Convention is to put all global usings in a single file named GlobalUsings.cs at the project root. Teams use this as the "dependency manifest" — one place to see all third-party namespaces in use.
ERP.Enrollment/GlobalUsings.cs
3
global using static
You can also do global using static System.Console — then you can write WriteLine("hi") without the Console. prefix everywhere. Common in test projects with using static Assert.
global using static System.Console;

Build Outputs Explained
1
obj/ — intermediates
The obj/ folder holds intermediate build files — generated code, NuGet dependency graphs, compile caches. Never commit to Git. Delete it with dotnet clean if you have weird build errors.
dotnet clean # removes obj/ and bin/
2
bin/ — your assembly
The .dll in bin/ is your compiled IL assembly. The .exe is just a tiny launcher that calls into the DLL. The .pdb holds debug symbols — line numbers and variable names for the debugger.
.dll = IL code | .pdb = debug info
3
Debug vs Release
Debug builds include debug info, disable optimisations, and are slower. Release builds are optimised, smaller, and have no debug symbols. Always test performance with Release builds — Debug is misleadingly slower.
dotnet build -c Release

06 · Watch Out
Common Mistakes — and How to Fix Them
⚠️
Mistakes Beginners Make in M01
❌
Picking ".NET Framework" instead of ".NET 8" in the New Project dialog. They look similar but are completely different platforms. .NET Framework is Windows-only and legacy. .NET 8 is modern, cross-platform, and LTS. Fix: Look for "(Long Term Support)" label in the framework dropdown. Always pick net8.0.
❌
Using double for money values: double fee = 15000.50; This causes floating-point rounding errors. 0.1 + 0.2 = 0.30000000000000004 in double. Fix: Always use decimal for money: decimal fee = 15000.50m; (note the 'm' suffix)
❌
Closing the console window immediately: You run the app with F5 and the window disappears before you can read the output. Fix: Use Ctrl+F5 (Run Without Debugging) — VS keeps the console open. Or add Console.ReadKey(); at the end.
❌
Forgetting the namespace after creating a class: You create Student.cs in a Models folder but CS0246 error says type not found. The namespace in Student.cs doesn't match what you used in Program.cs. Fix: Use file-scoped namespaces: namespace ERP.Enrollment.Models; (note the semicolon, no braces)
❌
CS8618 warning on string properties: With Nullable=enable, non-nullable string properties must be initialised. public string Name { get; set; } gets a warning. Fix: Add a default initialiser: public string Name { get; set; } = "";
