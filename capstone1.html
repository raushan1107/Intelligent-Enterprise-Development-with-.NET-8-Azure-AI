2 Add a custom exception and a validation method

Add the custom exception class at the bottom of the file, outside the Capstone1 class:

Custom exception — bottom of Capstone1.cs 
class RegistrationException : Exception
{
    public RegistrationException(string message) : base(message) { }
}

Now add a validation method inside the Capstone1 class:

Validation method 
static void Validate(string name, int age, int marks)
{
    if (name == "" || name == "Unknown")
        throw new RegistrationException("Name cannot be empty.");

    if (age < 16 || age > 80)
        throw new RegistrationException("Age must be between 16 and 80.");

    if (marks < 0 || marks > 100)
        throw new RegistrationException("Marks must be between 0 and 100.");
}
3 Parse and validate inside Run()

After reading the inputs, parse and validate them. Wrap everything in try/catch:

Parse, validate, and handle errors in Run() 
int age   = int.TryParse(ageInput,   out int a) ? a : 0;
int marks = int.TryParse(marksInput, out int m) ? m : 0;

try
{
    Log("INFO", "Starting registration for " + name);
    Validate(name, age, marks);
    Log("INFO", "Age validation passed");
    Log("INFO", "Marks validation passed");
}
catch (RegistrationException ex)
{
    Log("ERROR", "Validation failed: " + ex.Message);
    return;   // stop — don't continue with invalid data
}
💡

The ? : on the TryParse lines is a ternary expression — it means: if TryParse succeeds use the parsed value, otherwise use 0. It is shorthand for an if/else.

4 Add the Log helper method

Add this inside the Capstone1 class:

Log() helper 
static void Log(string level, string message)
{
    string time = DateTime.Now.ToString("HH:mm:ss");
    Console.WriteLine("[" + level + "] " + time + " - " + message);
}

Run the program. Try entering an invalid age like -5 and confirm the error is caught and printed.

PHASE 3

Fee Calculator

Modules 4 (methods + overloading)
1 Add the fee calculation methods

Add these two overloads inside the Capstone1 class. The first uses a default GST of 18%. The second lets you pass a custom rate.

Overloaded fee methods 
// Overload 1 — uses default 18% GST
static double CalculateFee(double baseFee)
{
    return baseFee + (baseFee * 0.18);
}

// Overload 2 — custom tax rate
static double CalculateFee(double baseFee, double taxRate)
{
    return baseFee + (baseFee * taxRate);
}
2 Call it from Run()

After the validation try/catch block, add:

Calculate and store the fee 
double baseFee  = 29999;
double totalFee = CalculateFee(baseFee);   // uses the 18% overload
double gstAmt   = totalFee - baseFee;
PHASE 4

Grade & Summary

Modules 2, 3, 4
1 Add a GetGrade method
GetGrade — returns A, B, C, or F 
static string GetGrade(int marks)
{
    if (marks >= 90) return "A";
    if (marks >= 75) return "B";
    if (marks >= 50) return "C";
    return "F";
}
2 Add a PrintSummary method
PrintSummary — displays everything 
static void PrintSummary(string name, int age,
                           int marks, double baseFee,
                           double gst, double total)
{
    string grade = GetGrade(marks);

    Console.WriteLine();
    Console.WriteLine("--- Registration Summary ---");
    Console.WriteLine("Name    : " + name);
    Console.WriteLine("Age     : " + age);
    Console.WriteLine("Marks   : " + marks);
    Console.WriteLine("Grade   : " + grade);
    Console.WriteLine("Course  : .NET 8 and Azure AI (11 days)");
    Console.WriteLine("Base Fee: " + baseFee);
    Console.WriteLine("GST 18% : " + gst);
    Console.WriteLine("Total   : " + total);
}

Add this after the fee calculation lines:

Run the program with valid input. You should see the full summary.

PHASE 5

Wrap Run() in a top-level try/catch

Module 5 — final polish
1 Protect the whole Run() with a catch-all

Any unexpected exception that slips past your validation will now be caught here instead of crashing the program with a raw error screen.

Final Run() structure 
public static void Run()
{
    PrintHeader();

    // Read input
    Console.Write("Enter your name   : ");
    string name = Console.ReadLine() ?? "Unknown";

    Console.Write("Enter your age    : ");
    string ageInput = Console.ReadLine() ?? "0";

    Console.Write("Enter marks (0-100): ");
    string marksInput = Console.ReadLine() ?? "0";

    int age   = int.TryParse(ageInput,   out int a) ? a : 0;
    int marks = int.TryParse(marksInput, out int m) ? m : 0;

    Console.WriteLine();

    try
    {
        Log("INFO", "Starting registration for " + name);
        Validate(name, age, marks);
        Log("INFO", "Age validation passed");
        Log("INFO", "Marks validation passed");

        double baseFee  = 29999;
        double totalFee = CalculateFee(baseFee);
        double gstAmt   = totalFee - baseFee;

        PrintSummary(name, age, marks, baseFee, gstAmt, totalFee);
        Log("INFO", "Registration complete for " + name);
    }
    catch (RegistrationException ex)
    {
        Log("ERROR", "Validation failed: " + ex.Message);
    }
    catch (Exception ex)
    {
        Log("CRITICAL", "Unexpected error: " + ex.Message);
    }
}
💡

Notice how the flow reads almost like plain English: validate → calculate → print → log done. Each method does one thing. That is good code design — and it is what the refactoring step in Module 4 was building toward.
