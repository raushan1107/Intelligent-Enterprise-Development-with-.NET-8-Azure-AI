01 · Concepts
The C# Type System — What You Must Know
📦
Value Types — Copy on Assign
Value types store data directly on the stack. When you write b = a, b gets an independent copy. Changing b never affects a. Built-ins: int, long, decimal, bool, char, double, DateTime, any struct.
int a = 10; int b = a; // independent copy b = 20; Console.WriteLine(a); // 10 — unchanged ✓
🔗
Reference Types — Share on Assign
Reference types store a pointer to a heap object. When you write s2 = s1, both variables point to the same object. Mutating through s2 changes what s1 sees too. This is the most common source of ERP data bugs.
var s1 = new Student { Name = "Alice" }; var s2 = s1; // copies the POINTER s2.Name = "Bob"; Console.WriteLine(s1.Name); // "Bob" 🐛
💰
decimal — Always for Money
ALWAYS use decimal for fees, salaries, tax. double uses binary floating-point — 0.1 + 0.2 ≠ 0.3 exactly. This isn't a bug; it's how binary fractions work. decimal uses base-10 with 28–29 significant digits — exact for currency.
double d = 0.1 + 0.2; Console.WriteLine(d); // 0.30000000000000004 ❌ decimal m = 0.1m + 0.2m; Console.WriteLine(m); // 0.3 ✓ (m = decimal)
❓
Nullable Types — Explicit Null
With Nullable=enable, string cannot be null. Use string? to explicitly allow it. Works for value types too: int? is Nullable<int>. Use ?? for fallback, ?. to safely navigate null chains without throwing.
string? alias = null; int? score = null; string display = alias ?? "Unknown"; // fallback int? len = alias?.Length; // null, no throw int final = score ?? 0; // 0 if null
🔄
Type Conversion — Safe vs Unsafe
Implicit (widening): always safe, no data loss (int → long). Explicit cast: truncates, may lose data ((int)3.9 = 3, not 4). TryParse: safe parsing of strings — never use Parse() on user input (it throws).
int i = 42; long l = i; // implicit — safe double d = 3.9; int n = (int)d; // explicit → 3 (truncated!) // Always TryParse on user input: if (int.TryParse(input, out int age)) { ... }
🎯
string — Immutable Reference Type
string is a reference type but behaves like a value because strings are immutable — they can never be changed after creation. Every "change" creates a new string. Use StringBuilder for repeated concatenation inside loops.
string x = "Hello"; string y = x; // both point to "Hello" x = "World"; // NEW string — y unchanged Console.WriteLine(y); // "Hello" ✓ // Loop concatenation: use StringBuilder var sb = new StringBuilder(); foreach (var n in names) sb.Append(n);

ValueVsReference.cs
// ── VALUE TYPE: independent copy ── int a = 10; int b = a; // stack: new slot, value=10 b = 20; Console.WriteLine(a); // 10 — not affected ✓ Console.WriteLine(b); // 20 // ── REFERENCE TYPE: shared pointer ── var s1 = new Student { Name = "Alice" }; var s2 = s1; // copies POINTER, not object s2.Name = "Bob"; Console.WriteLine(s1.Name); // "Bob" ← bug! 🐛 // ── FIX A: new object ── var s3 = new Student { Name = s1.Name, Id = s1.Id }; s3.Name = "Charlie"; Console.WriteLine(s1.Name); // "Bob" — s1 untouched ✓ // ── FIX B: record with-expression ── public record StudentRecord(int Id, string Name); var r1 = new StudentRecord(1, "Alice"); var r2 = r1 with { Name = "Charlie" }; // new copy Console.WriteLine(r1.Name); // "Alice" ✓
Key Points
1
int b = a → two slots
Two independent stack slots. After b = 20, stack looks like: [a=10][b=20]. They will never share state again.
Stack: [a=10] [b=20] (independent)
2
var s2 = s1 → same address
Stack has two pointer slots — both holding address 0x5800. There is only ONE Student object on the heap. s2.Name = "Bob" modifies the shared object.
Stack: [s1→0x5800] [s2→0x5800]
3
record with { } → new object
with is a non-destructive copy — it creates a new object with all the same values, then applies your overrides. The original is never touched. Requires the type to be a record.
r1 with { Name = "X" } → new record
NullSafety.cs
// Nullable=enable in .csproj string name = "Alice"; // cannot be null string? alias = null; // ? = explicitly nullable int? score = null; // Nullable<int> // ── ?. NULL-CONDITIONAL ── // Safe chain: if ANY part is null → result is null string? city = student?.Address?.City; // student=null → city=null (no NullReferenceException) // student.Address=null → city=null // both set → city = "Mumbai" // ── ?? NULL-COALESCING ── string display = alias ?? "Unknown"; int final = score ?? 0; // ── ??= NULL-COALESCING ASSIGNMENT ── alias ??= "Default"; // assigns ONLY if currently null // ── ! NULL-FORGIVING (use sparingly) ── string forced = alias!; // "trust me, not null" // ⚠ Only use when YOU are certain // ── TryParse — never Parse() on user input ── string? raw = Console.ReadLine(); if (int.TryParse(raw, out int age)) Console.WriteLine($"Valid age: {age}"); else Console.WriteLine("Not a valid number"); // No FormatException — ever.
Null Safety Patterns
1
?. short-circuits the chain
If any operand is null, the whole expression returns null immediately. No exception. The result type is always nullable. Common in EF Core navigation: order?.Customer?.Email.
a?.b?.c → null if a or b is null
2
?? chains infinitely
The right side evaluates lazily — only when the left is null. Chain multiple: a ?? b ?? c ?? "default". Use ??= for lazy initialisation patterns.
config ?? env ?? "default"
3
TryParse vs Parse
int.Parse("abc") throws FormatException. int.TryParse("abc", out int n) returns false and sets n = 0. Always use TryParse on any data you don't fully control.
TryParse → false (safe) Parse → throws (dangerous)
TypeConversion.cs
// ── IMPLICIT (widening — zero data loss) ── int i = 100; long l = i; // int fits in long ✓ double d = i; // int fits in double ✓ // ── EXPLICIT (narrowing — may lose data) ── double pi = 3.14159; int piInt = (int)pi; // 3 — TRUNCATED, not rounded // Use Math.Round() first if you need rounding: int piRound = (int)Math.Round(pi); // 3 int piConv = Convert.ToInt32(pi); // 3 (banker's round) // ── CHECKED — throw on overflow ── int max = int.MaxValue; // 2,147,483,647 int wrapped = max + 1; // -2147483648 (silent!) int safe = checked(max + 1); // throws OverflowException // ── NUMBER LITERALS ── int million = 1_000_000; // underscores for readability int flags = 0b1010_1100; // binary literal int hex = 0xFF_AA; // hex literal decimal fee = 85_000.00m; // m suffix = decimal float ratio = 0.5f; // f suffix = float long bigId = 9_000_000_000L; // L suffix = long
Conversion Rules
1
Cast truncates, not rounds
(int)3.9 = 3. The .9 is simply dropped. This catches many people off-guard. If you need rounding: use Math.Round() first. Convert.ToInt32() uses banker's rounding (round half to even).
(int)3.9 = 3 (truncate) (int)3.5 = 3 (truncate) Convert.ToInt32(3.5) = 4 (round)
2
checked catches overflow
By default, integer arithmetic wraps silently on overflow. int.MaxValue + 1 = -2,147,483,648. The checked keyword makes this throw instead. Critical for financial code where overflow = data corruption.
checked(int.MaxValue + 1) → throws
3
Suffixes lock the type
Without suffix, literals default to int or double. Use m for decimal, f for float, L for long. Missing the m on a decimal literal creates a double — then an implicit narrowing conversion happens.
15000 → int 15000.0 → double 15000m → decimal ← correct for fees
ErpFees.cs — Real ERP Context
// ── ERP FEE MODEL — correct types ── public class StudentFee { public int StudentId { get; set; } public string CourseName { get; set; } = ""; public decimal BaseFee { get; set; } // ← decimal public decimal? Scholarship { get; set; } // ← nullable public DateTime DueDate { get; init; } // ← init-only public decimal EffectiveFee => BaseFee - (Scholarship ?? 0m); // computed public decimal TotalWithGST => EffectiveFee * 1.18m; // 18% GST } // ── USAGE ── var fee = new StudentFee { StudentId = 101, CourseName = "Full Stack .NET 8", BaseFee = 85_000m, Scholarship= 10_000m, // optional DueDate = new DateTime(2025, 6, 1) }; Console.WriteLine($"Base fee : {fee.BaseFee:C0}"); Console.WriteLine($"Scholarship: {fee.Scholarship:C0}"); Console.WriteLine($"Effective : {fee.EffectiveFee:C0}"); Console.WriteLine($"Total+GST : {fee.TotalWithGST:C0}");
ERP Type Decisions
1
decimal for every fee
BaseFee, Scholarship, GST — all decimal. Never mix with double in calculations. If a double sneaks in, C# will implicitly convert decimal → double and you'll get rounding errors silently.
decimal fee = 85_000m; // always
2
decimal? for optional amounts
Scholarship is decimal? because not every student has one. Using 0 as "no scholarship" is ambiguous. Null is unambiguous: it means the field simply doesn't apply. The ?? 0m in the computed property handles it cleanly.
Scholarship ?? 0m // fallback to zero
3
init-only for immutable dates
init means the property can only be set during object construction (in the object initialiser or a constructor). After that, it's read-only. DueDate should never change once a fee record is created.
DueDate { get; init; } // set once
PatternMatching.cs
// ── TYPE PATTERN ── object item = GetEnrollmentItem(); if (item is Student s) Console.WriteLine($"Student: {s.Name}"); else if (item is Course c) Console.WriteLine($"Course: {c.Code}"); // ── RELATIONAL PATTERN ── string category = score switch { >= 90 => "Distinction", >= 75 and < 90 => "First Class", >= 60 => "Second Class", >= 40 => "Pass", _ => "Fail" }; // ── PROPERTY PATTERN ── bool atRisk = student is { GPA: < 2.0m, IsActive: true }; // ── NULL CHECK PATTERN ── if (student is not null) Process(student); // ── RANGE VALIDATION ── if (age is >= 15 and <= 60) Enroll(student); else throw new ArgumentException("Age must be 15–60");
Pattern Matching
1
Type pattern is and cast
if (item is Student s) checks the type AND gives you a typed variable s in one step. No separate cast. If item is null or a different type, the block is skipped entirely — no exception.
is Student s → typed + null safe
2
Switch expression → value
Switch expressions return a value — they're expressions, not statements. Every arm must return the same type. The compiler verifies exhaustiveness — if you miss a case you get a warning. The _ discard covers everything else.
score switch { ... _ => "Fail" }
3
Property pattern — no if-else chains
Test multiple properties inline without nesting. { GPA: < 2.0m, IsActive: true } replaces an if-chain. Can be nested: { Address: {{ City: "Mumbai" }} }.
{ GPA: < 2.0m, IsActive: true }

07 · Watch Out
Common Mistakes in M02
⚠️
Mistakes with Types That Bite Later
❌
Using double for money: double fee = 15000.50 + 500.30; gives 15500.799999999998 — a tiny error that compounds over hundreds of transactions. Fix: Always decimal. decimal fee = 15000.50m + 500.30m; // exact: 15500.80
❌
Expecting var s2 = s1 to clone the object: Beginners assume assignment copies the object. It doesn't — it copies the reference. Both variables point to the same Student. Fix: var s2 = new Student { Name = s1.Name, Id = s1.Id }; // truly independent
❌
Using int.Parse() on Console input: If the user types letters, int.Parse(Console.ReadLine()!) throws an unhandled FormatException and crashes the app. Fix: if (int.TryParse(Console.ReadLine(), out int n)) { use n } else { show error }
❌
Forgetting the 'm' suffix on decimal literals: decimal fee = 15000.5; — the literal 15000.5 is a double. C# implicitly converts but gives a precision warning. Fix: decimal fee = 15000.5m; // 'm' suffix = decimal literal
❌
Accessing a nullable without checking first: int? score = null; Console.WriteLine(score.Value); throws InvalidOperationException. Fix: Console.WriteLine(score ?? 0); // or: if (score.HasValue) { ... }
